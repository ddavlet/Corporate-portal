# Колберг — обзор продукта

> Краткая характеристика платформы как единого целого: что это, из чего состоит и что на ней можно реализовать.

## Что это

**Колберг** — мультитенантная SaaS-платформа для управления финансовыми операциями компании. Один движок, на котором каждая компания (tenant) живёт на своём поддомене (`company.kolberg.uz`), со своим набором включённых модулей, ролей и прав доступа.

Суть продукта — навести порядок в деньгах компании: заявки на оплату, бюджеты, зарплаты, инвестиции и отчётность, с мобильным согласованием платежей прямо в Telegram.

Если в двух словах — это **финансовый бэк-офис компании как конструктор**: включаешь нужные модули, настраиваешь формы заявок и цепочки согласований под свои процессы, а сотрудники работают и подтверждают платежи хоть из браузера, хоть из Telegram.

## Ядро системы — заявки на оплату

Заявки (`requests`) — сердце платформы и **единственный источник правды по расходам**. Любая трата сначала проходит как заявка с настраиваемой формой и цепочкой согласований. Согласовать можно из веба или мобильно — кнопками прямо в Telegram.

Кассовые, банковские и карточные движения существуют отдельно как «сырые» бухгалтерские ленты и нужны лишь для сверки: у каждого расхода должна быть своя заявка.

## Модули

| Модуль | Что закрывает |
|---|---|
| `requests` | Заявки на оплату, формы, цепочки согласований, авто-заявки |
| `cash` | Касса: приходы и расходы наличными |
| `bank` | Банковские счета: приходы и расходы |
| `corporate_card` | Корпоративные карты: движения по картам |
| `wallets` | Кошельки и балансы по каналам |
| `budgets` | Бюджеты, лимиты, заявки на трату в рамках бюджета |
| `payroll` | Зарплата: расчётные документы по сотрудникам |
| `investments` | Инвесторы, графики выплат, возвраты + публичная страница графика |
| `clients_debt` | Долги клиентов |
| `contracts` | Контракты |
| `vendors` | Справочник поставщиков |
| `reports` | Отчётность: P&L и cashflow |
| `tasks` | Задачи |
| `notes` | Заметки |
| `feedback` | Обратная связь |

## Доступ — модульно-ролевой

Эффективный доступ к функции = выполнены **все** условия одновременно:

1. пользователь авторизован;
2. состоит в активном membership внутри tenant;
3. модуль включён для этой компании (`TenantModuleConfig`);
4. роль пользователя допущена к ключу модуля.

Это позволяет «собрать» одну и ту же платформу под конкретную компанию: включить только нужные модули и раздать права (admin, director, accountant и т.д.). Например, финансовая запись в кошельки доступна только `admin/accountant`, часть настроек заявок — только `admin/director`.

## Точки интеграции (намеренно только три)

- **Web UI** — React SPA для веб-пользователей (`/app`).
- **Telegram WebApp** — мобильный интерфейс и согласование платежей кнопками (`/tg/*`).
- **n8n** — автоматизация: уведомления, пайплайны данных, авто-заявки.

## Технический стек

- **Backend** — Django REST API (Python 3.12), tenant-aware, JWT-аутентификация.
- **Frontend** — React SPA (Ant Design), локаль `ru_RU`, таймзона `Asia/Tashkent`.
- **Telegram-шлюз** — FastAPI (uvicorn).
- **БД** — PostgreSQL 17 с pgvector.
- **Инфраструктура** — Docker Compose, Traefik (reverse proxy + TLS), деплой через Makefile.

Мультитенантность — по поддоменам: backend определяет tenant из заголовка `Host` на каждом запросе.
