# План: комментарии к заявкам

## 1. Цель задачи

Добавить в карточку заявки **независимую** систему комментариев. Комментарий — это свободный текст, который никак не связан с действием одобрения или отклонения. Назначение:

- задать вопрос («на какой бюджет относим?»);
- предложить уточнение или задачу («приложите счёт перед оплатой»);
- оставить пояснение к ходу согласования;
- просто прокомментировать ситуацию.

Решения «Одобрить / Отклонить» остаются **полностью отдельным** потоком и не затрагиваются этой задачей. Комментарий — это опциональное действие, которое можно совершить до, после или вместо решения.

---

## 2. Что добавляем

1. **Блок «Комментарии» в карточке заявки** (портал) — лента комментариев, новые сверху.
2. **Форма добавления комментария** в портале — кнопка «Добавить», inline-textarea, кнопки «Отправить / Отмена».
3. **Кнопка «💬 Оставить комментарий»** в каждом Telegram-сообщении бота — прямая ссылка на Mini App, необязательная, появляется на всех этапах согласования.
4. **Плашка «💬 N комментариев»** в Telegram — при каждом обновлении сообщения согласующий видит актуальное количество и тизер последнего комментария.
5. **Индикатор комментариев** в портале рядом с кнопками решения — согласующий видит, что есть обсуждение, до того как принять решение.

---

## 3. Хранение данных

### Новая таблица: `request_comments`

| Поле | Тип | Описание |
|---|---|---|
| `id` | автоинкремент | первичный ключ |
| `request_id` | FK → `requests` | к какой заявке; при удалении заявки каскадно удаляется |
| `created_by_id` | FK → `users` | автор; защищён от удаления |
| `body` | текст | сам комментарий (лимит 4000 символов на уровне API) |
| `created_at` | дата/время | когда оставлен |

- Тенант не дублируется — берётся через `request.tenant`, как уже сделано для `Approval`.
- Индекс: `(request_id, created_at)` — быстрая выборка ленты комментариев по заявке.
- Дополнительных таблиц состояния не требуется — Telegram-интеграция реализована через прямую ссылку без промежуточного состояния на сервере.

## 4. Архитектура: бэкенд

**Где живёт:** модуль `apps/modules/requests/` — рядом с `Request` и `Approval`. Отдельный модуль не создаём.

**Новые сущности:**
- Модель `RequestComment` (таблица описана в §3).
- Сериализатор `RequestCommentSerializer` — отдаёт `id`, `body`, `created_at`, `created_by`, `created_by_full_name`.
- Поле `comments` в ответе `GET /api/requests/<id>/` — массив комментариев, новые сверху.

**Новые HTTP-эндпоинты:**
- `GET /api/requests/<id>/comments/` — список комментариев.
- `POST /api/requests/<id>/comments/` — создать комментарий, тело запроса: `{ "body": "..." }`.

**Контроль доступа:** те же права, что и для просмотра заявки. Комментировать может любой, кто видит заявку — заявитель, любой из согласующих, админ тенанта. Дополнительной ролевой проверки нет.

**Валидация:** `body` после trim не пустой; длина ≤ 4000 символов.

**Обновление Telegram при новом комментарии:** после сохранения комментария (из портала или через Mini App) бэкенд вызывает существующий механизм `refresh_request_messages` — он обновляет активные сообщения текущего этапа, и согласующий видит актуальную плашку «💬 N комментариев».

## 5. Архитектура: Telegram-интеграция

### Принцип: прямая ссылка на Mini App (тот же подход, что у payment webapp)

В системе уже реализован паттерн «кнопка в Telegram открывает Mini App»: когда шаг выплаты имеет тип `webapp`, кнопка «Подтвердить выплату» — это прямая ссылка вида `t.me/<bot>/<app>?startapp=<approval_id>`. Telegram открывает Mini App автоматически, без колбэка на сервер.

Для комментариев используем **точно такой же паттерн**:

- Кнопка «💬 Оставить комментарий» — прямая ссылка `t.me/<bot>/<app>?startapp=req_<request_id>`.
- Telegram открывает Mini App; параметр `startapp` содержит ID заявки.
- Mini App показывает форму с textarea.
- Пользователь пишет комментарий и нажимает «Отправить».
- Mini App вызывает `POST /api/requests/<id>/comments/` — тот же эндпоинт, что и из портала.
- Комментарий сохраняется, Mini App закрывается.

### Ключевое отличие от payment webapp

| | Payment webapp | Comment Mini App |
|---|---|---|
| `startapp` содержит | `approval_id` | `req_<request_id>` |
| Привязка | К конкретной строке `Approval` | К заявке в целом |
| На каких этапах | Только шаг типа `payment` + mode `webapp` | На **всех** этапах согласования |
| Эндпоинт | `/api/requests/approvals/payment-webapp/confirm/` | `/api/requests/<id>/comments/` |

Параметр `startapp` содержит `request_id`, а не `approval_id`, потому что комментарий принадлежит заявке, а не конкретному шагу.

### Расположение кнопки в сообщении

Кнопка «💬 Оставить комментарий» добавляется **на отдельной строке** под кнопками «Одобрить / Отклонить» в функции `build_approval_message`. Появляется на всех этапах согласования безусловно.

```
[✅ Одобрить]  [❌ Отклонить]
[💬 Оставить комментарий    ]
```

### Плашка комментариев в тексте сообщения

Если комментарии есть — в конец сообщения добавляется блок:

```
💬 Комментариев: 3
Последний от Иван Иванов:
«Нужен счёт-фактура перед оплатой, уточните у бухгалтера…»
```

- Тизер — не более 120 символов последнего комментария.
- Если комментариев нет — блок не добавляется, сообщение не меняется.

## 6. Архитектура: фронтенд

### Портал

**Новый компонент `RequestCommentsSection`** — встраивается в страницу заявки между телом заявки и списком этапов согласования.

**Новые функции в API-клиенте:**
- `createRequestComment(requestId, body)`.

**Расширение типов:**
- `RequestDetail.comments: RequestComment[]` — комментарии уже приходят в ответе детали заявки, отдельный запрос не нужен.

### Mini App (Telegram)

**Новая страница** по образцу существующего `TgPaymentConfirmPage`:

- Читает `startapp` параметр → извлекает `request_id`.
- Показывает заголовок «Комментарий к заявке #N».
- Textarea для ввода (с ограничением 4000 символов и счётчиком).
- Кнопка «Отправить».
- После успешной отправки — показывает «✅ Комментарий сохранён», закрывает Mini App.


### Длинные комментарии

- По умолчанию — обрезка до 3 строк (CSS `line-clamp`).
- Если текст длиннее — кнопка «Показать полностью».
- При раскрытии — кнопка меняется на «Свернуть».
- Раскрытие происходит inline, без модального окна.

### Индикатор у кнопок решения

В блоке «На рассмотрении» (над кнопками «Одобрить / Отклонить»):

```
💬 3 комментария  [перейти к обсуждению]
[ Одобрить ]
[ Отклонить ]
```

Клик — плавная прокрутка к блоку «Комментарии».

## 8. Ключевые дизайн-решения

| Решение | Обоснование |
|---|---|
| Комментарии — отдельная сущность, не поле Approval | Не привязан к шагу или решению; любой может оставить любое количество в любое время |
| Тот же паттерн прямой ссылки, что у payment webapp | Не нужна новая инфраструктура, паттерн уже проверен в production |
| `startapp = req_<request_id>`, не `approval_id` | Комментарий принадлежит заявке, а не конкретному шагу |
| Кнопка на всех этапах, не только payment | Комментировать нужно на любом этапе |
| Один эндпоинт `POST /api/requests/<id>/comments/` и для портала, и для Mini App | Нет дублирования логики |
| Inline-форма в портале | Меньше кликов, пользователь не теряет контекст |
| Сортировка «новые сверху» | Стандарт для лент обсуждений (Jira, Trello, Slack) |
| `refresh_request_messages` при новом комментарии | Согласующий видит актуальное обсуждение без дополнительных действий |
| Без редактирования / удаления в v1 | Минимальный scope; аудит обсуждения остаётся неизменным |

---

## 9. Что НЕ входит в эту итерацию

- Редактирование и удаление комментариев.
- Упоминания пользователей (`@username`) и адресные уведомления.
- Реакции и эмодзи.
- Вложения внутри комментария.
- Ввод комментария свободным текстом прямо в Telegram-чате (без Mini App).
- Комментарии на cash- / bank-расходах (можно обобщить позже при необходимости).

---

## 10. Этапы внедрения

1. **БД и API** — модель `RequestComment`, миграция, сериализатор, эндпоинты `GET/POST /api/requests/<id>/comments/`, поле `comments` в ответе детали заявки.
2. **Telegram: отображение** — плашка «💬 N комментариев» в тексте сообщения; вызов `refresh_request_messages` при создании комментария.
3. **Telegram: кнопка и Mini App** — добавить кнопку прямой ссылки в `build_approval_message`; новая страница Mini App с формой; авторизация через Telegram initData (тот же механизм, что у payment Mini App).
4. **Фронтенд (портал)** — компонент `RequestCommentsSection`, inline-форма, поведение «Показать полностью / Свернуть», индикатор у кнопок решения.
5. **Ручная проверка сценариев:**
   - написать комментарий из портала → плашка появилась в Telegram у согласующего следующего этапа;
   - написать комментарий через Mini App → виден в портале мгновенно;
   - длинный комментарий корректно сворачивается и раскрывается;
   - пустой и слишком длинный комментарий отклоняются сервером.
